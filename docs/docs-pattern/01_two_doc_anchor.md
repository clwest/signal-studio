# The Two-Doc Anchor Pattern

Two docs carry the weight of every quantitative claim about your platform. Every other doc either points at these or drifts. The pattern:

| File | Role | Who writes it |
|---|---|---|
| `docs/<NAME>_WHAT_IT_IS.md` | **Narrative / conceptual** — one-pager that explains what the system is, how the layers fit, what's unique, and where to look | Humans + AI (by hand) |
| `docs/<NAME>_INVENTORY.md` | **Runtime-derived** — regenerable snapshot of what currently exists (counts, lists, tables) | Management command (auto) |

In this repo: `PLATFORM_WHAT_IT_IS.md` + `PLATFORM_INVENTORY.md`. On a new app, pick a name (e.g. `APPNAME_WHAT_IT_IS.md`) and match it.

---

## Why two docs, not one

One doc that tries to be both always becomes neither:
- Narrative docs with inline exact counts go stale within days.
- Pure inventory dumps have no context — a reader can't tell why anything matters.

Splitting lets each doc do its job:
- **`WHAT_IT_IS.md`** uses `~` approximations or "live counter" annotations for anything that moves. Focus on *structure and intent*.
- **`INVENTORY.md`** is regenerated on command. Exact counts, exact lists, exact filenames. No narrative.

---

## `WHAT_IT_IS.md` — Structure that worked

Borrow directly from `docs/PLATFORM_WHAT_IT_IS.md`:

```markdown
---
title: "<App> — What It Actually Is"
status: active
companion_doc: <APP>_INVENTORY.md
---

# <App> — What It Actually Is

> **Read-order note:** this doc is the narrative companion to
> <APP>_INVENTORY.md. The inventory is runtime-derived and regenerable.
> It tells you *what exists right now*. This doc tells you *what it all
> is, why it exists, and how it fits together*.

## TL;DR — One Sentence
<one-sentence pitch>

## Table of Contents
1. The Core Idea
2. The Layered Architecture
3. The Unique Stuff (The IP)
4. Current State Honesty
5. How to Navigate
6. Glossary
```

**Section guidelines:**
- **Core Idea** — 3–6 sentences. What this system is, at the level your grandma could understand.
- **Layered Architecture** — numbered layers (data → agents → pipeline → UI → infra). Each layer has one paragraph + one table.
- **Unique Stuff** — bullets. What's distinctive vs off-the-shelf. This section is what you'd show an investor.
- **Current State Honesty** — two sub-sections: "working well" and "stale or drifting." Never hide drift.
- **How to Navigate** — the six docs to know + the three commands to remember.
- **Glossary** — every term that's specific to your platform. Alphabetized. One line each.

**Length target:** 300–500 lines. Long enough to be load-bearing, short enough to read in a single session.

## `INVENTORY.md` — Structure that worked

Regenerated by a management command. In this repo: `python manage.py generate_platform_inventory`.

**Required properties:**
1. **Regenerable** — one command, never hand-edited.
2. **Deterministic** — same code → same output (sort everything).
3. **Sectioned** — TOC + 15–25 sections, each independently readable.
4. **Runtime-derived** — counts come from `AGENT_MAP`, `apps.get_models()`, `CELERY.tasks`, etc. — never hard-coded.
5. **Fast** — under 5 seconds to regenerate. If it's slow, nobody runs it.

**Sections that worked on this platform** (adapt to yours):
- Executive Summary (all counts at a glance)
- Agents / Services / Tasks / Models
- Routes / Views / Management Commands
- Integrations / External APIs
- Infrastructure (workers, queues, DBs)
- Code Statistics (LOC per directory)
- Doc-vs-Reality Verifier State (rollup of the verifier output)

The "Verifier State" section is the payoff — running the inventory also tells you which docs are out of sync.

## The One Rule That Ties It Together

> **When two docs disagree, `INVENTORY.md` wins.**

Every other doc in the repo — `CLAUDE.md`, the topic docs, the audit dossiers — gets a header block:

```markdown
> **Source of truth for numbers:** PLATFORM_WHAT_IT_IS.md (narrative)
> + PLATFORM_INVENTORY.md (runtime-derived, regenerable). When this doc
> disagrees with either, PLATFORM_INVENTORY wins.
```

This lets you stop chasing drift in 100 docs. The canonical pair is maintained; everything else can drift and the reader knows where to look.

---

## When to use this pattern

- Any codebase with >20 docs
- Any codebase touched by AI pair programming across multiple sessions
- Any team where one person wrote the docs and four people need to trust them

**When to skip it:** single-author project with <10 docs total. The pattern is overhead below that scale.

---

## Failure modes to watch

- **Multiple "source of truth" docs for *the same kind of claim*** — if you find yourself writing a 3rd doc that holds quantitative facts, stop. The quantitative anchor pair is two docs, not three.
- **Inventory that hand-edits** — the moment you open `INVENTORY.md` in an editor, the pattern is dead. Regenerate only.
- **`WHAT_IT_IS.md` with exact counts** — if someone greps for a number, they find it in two places, and one is wrong. Use `~` or "live counter" annotations in the narrative doc.

---

## Complementary docs, not replacements: `PIPELINE.md`, `BEHAVIOR_LAYER.md`, `TRANSLATION_LAYER.md`

For projects with LLM, agent, task-runner, or queue-driven flows —
or any product that *speaks* in a defined voice — or any project
with multiple audiences who need to understand the same facts — the
two-doc anchor pair is necessary but not sufficient. The anchor pair
holds **what exists** and **what it is**. Three complementary docs
handle the rest:

- **`PIPELINE.md`** holds **how requests actually move through the
  system** — entry points, guard coverage, state surfaces, retrieval
  paths, post-processing order, and operational hazards.
- **`BEHAVIOR_LAYER.md`** holds **how responses sound, look, and
  respect prior turns** — voice / tone contract, UI source-of-truth
  rules ("don't restate rendered data in prose"), constraint
  preservation across follow-ups, decision-authority boundary
  between deterministic services and LLM phrasing, and small-model
  guard guidance.
- **`TRANSLATION_LAYER.md`** holds **how the same source-of-truth is
  explained to different audiences** — personas (builder /
  operator / executive / tester), translation modes (technical
  summary / business impact / executive brief / QA checklist), and
  truth-preservation rules ("same truth → different explanation,
  zero invention"). It reframes facts that already live in the
  other anchors; it does not introduce new ones.

None of these replace the anchor pair:

- They don't carry counts. INVENTORY still wins for numbers.
- They don't redefine the system. WHAT_IT_IS still owns the concept.
- They carry the *operational asymmetries*, *behavioral contracts*,
  and *audience contracts* that don't live naturally in either
  anchor.

All three are optional. Pick what you need:

| Project shape | What you probably need |
|---|---|
| Static site / single-script tool | WHAT_IT_IS + INVENTORY only |
| LLM-using API or agent system | + PIPELINE.md |
| Chat surface, voice agent, or persona-bearing UI | + BEHAVIOR_LAYER.md |
| Multi-stakeholder product (engineer + operator + executive + tester read the same docs) | + TRANSLATION_LAYER.md |
| All of the above | all five |

`context-kit init` scaffolds `docs/<APP>_PIPELINE.md`,
`docs/<APP>_BEHAVIOR_LAYER.md`, and `docs/<APP>_TRANSLATION_LAYER.md`
alongside the two-doc anchor. `context-kit doctor` warns when
project indicators suggest you'd benefit from each but the file is
missing. All three warnings are soft — never blocking — and
silently skipped on projects where they don't apply.

The translation layer is the youngest of the three, added after
stress testing showed that mature, multi-stakeholder projects
needed an explicit "same truth → different framing" contract. The
failure mode it catches: an assistant writing prose for an
executive, an operator, and a tester from the same source-of-truth,
but inventing different "progress" / "business impact" / "next
step" claims for each audience because no doc said *don't*.
